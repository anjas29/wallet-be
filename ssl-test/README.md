# SSL Pinning Test Harness — Ops Guide

Server-side companion to the Android certificate-pinning rollout. Lets QA/dev
drive cert scenarios (rotate, swap CA, force expiry) over HTTP and inspect the
live chain, so a pinned build can be validated against a real TLS handshake.

> **⚠️ Dangerous.** These routes rewrite the TLS identity the server presents.
> They are hard-gated behind `SSL_TEST_ENABLED=true` and are meant for the
> dedicated test VPS only. Never enable on a real production host.

---

## Architecture (this stack: Traefik terminates TLS)

Unlike a plain nginx+certbot box, **Traefik v3 terminates TLS and manages
Let's Encrypt** here. `wallet-be-nginx` only speaks plain HTTP on `:80` behind
Traefik. So cert control lives in **Traefik**, not nginx — we make Traefik serve
our swappable certs via its **dynamic file provider**:

```
POST /api/config/ssl-*        Laravel (wallet-be-app container, gated by SSL_TEST_ENABLED)
        │  Process:: runs the script in-container (no sudo, no docker socket)
        ▼
/opt/ssl-test/ssl-apply.sh    regenerates cert set + rewrites the dynamic YAML
        │  (writes to the bind-mounted /opt/ssl-test tree)
        ▼
/opt/ssl-test/dynamic/wallet-tls.yml   ← Traefik file provider (watch=true)
        │  file-write event → hot reload, no restart / signal / docker access
        ▼
Traefik serves the new leaf/chain for wallet.birchlabs.tech
```

**Why the privilege model differs from a classic www-data+sudo setup:** the app
runs as a root cli container with no host access, so it cannot `sudo` a host
script or signal Traefik. The boundary here is the `SSL_TEST_ENABLED` flag plus
test-box isolation. Keep this off any real prod host.

## Host layout (`/opt/ssl-test`, bind-mounted)

```
/opt/ssl-test/
├── ca/{a,b}/        ca.key, ca.crt, index.txt, serial, newcerts/   # manual OpenSSL CAs
├── sets/
│   ├── a/           fullchain.pem, privkey.pem, meta.json          # leaf under CA-A (pinned)
│   ├── b/           fullchain.pem, privkey.pem, meta.json          # leaf under CA-B (swap test)
│   └── c/           fullchain.pem, privkey.pem, meta.json          # real Let's Encrypt (swap-only)
├── dynamic/wallet-tls.yml     # Traefik file provider config (points at active set)
├── openssl-ca.cnf             # shared CA config for minute-precise issuance
├── ssl-apply.sh               # privileged mutator (runs in the app container)
├── ssl-info.sh                # read-only CLI inspector
└── bootstrap.sh               # one-time init
```

Copy this repo's `ssl-test/` contents to `/opt/ssl-test` on the VPS (the scripts
and `openssl-ca.cnf`); the `ca/`, `sets/`, and `dynamic/` trees are generated by
`bootstrap.sh`.

## Provisioning (one time, on the VPS)

### 1. Seed the tree

```bash
sudo mkdir -p /opt/ssl-test
sudo cp -r /opt/apps/wallet-be/src/ssl-test/. /opt/ssl-test/
sudo chmod +x /opt/ssl-test/*.sh
```

### 2. App container — mount the tree (+ acme.json for set c)

In `/opt/apps/wallet-be/docker-compose.yml`, add to the `app` service:

```yaml
  app:
    environment:
      APP_ENV: production
      APP_DEBUG: "true"
      SSL_TEST_ENABLED: "true"                 # arms the routes on THIS box only
      SSL_TEST_HOST: wallet.birchlabs.tech
    volumes:
      - ./src:/var/www/html
      - /opt/ssl-test:/opt/ssl-test            # rw: cert gen + dynamic YAML
      - /opt/traefik/acme.json:/letsencrypt/acme.json:ro   # ro: extract real LE cert (set c)
```

> `SSL_TEST_ENABLED` is read via `env()` → `config('ssltest.enabled')`. If the app
> is running cached config, run `php artisan config:clear` after changing it.

### 3. Traefik — add the file provider + mounts

In `/opt/traefik/docker-compose.yml`, add to the `traefik` service `command`:

```yaml
      - --providers.file.directory=/dynamic
      - --providers.file.watch=true
```

and to its `volumes`:

```yaml
      - /opt/ssl-test/dynamic:/dynamic:ro
      - /opt/ssl-test/sets:/certs:ro
```

### 4. Router — serve the file-provider cert (not the ACME resolver)

On the `nginx` service labels in `/opt/apps/wallet-be/docker-compose.yml`, replace
the cert-resolver label so the served cert comes from our store deterministically:

```yaml
    labels:
      - traefik.enable=true
      - traefik.http.routers.wallet.rule=Host(`wallet.birchlabs.tech`)
      - traefik.http.routers.wallet.entrypoints=websecure
      - traefik.http.routers.wallet.tls=true          # was: tls.certresolver=letsencrypt
      - traefik.http.services.wallet.loadbalancer.server.port=80
```

> Dropping the resolver stops Traefik auto-renewing the real LE cert for this
> host. That's fine for a test box — set `c` uses whatever cert is already in
> `acme.json` (valid ~90 days). To refresh it later, re-add the resolver label
> briefly (or run certbot) and re-run `ssl-apply.sh change c`.

### 5. Bring up + bootstrap

```bash
cd /opt/traefik && docker compose up -d
cd /opt/apps/wallet-be && docker compose up -d --build   # --build picks up openssl/jq in the Dockerfile
docker compose exec app bash /opt/ssl-test/bootstrap.sh
```

Verify Traefik is serving set `a`:

```bash
openssl s_client -connect wallet.birchlabs.tech:443 -servername wallet.birchlabs.tech </dev/null 2>/dev/null | openssl x509 -noout -issuer
# issuer=CN = Test Root CA A
```

## Endpoints

All require `auth:sanctum` + `throttle:6,1` and 404 unless `SSL_TEST_ENABLED=true`.

| Method | Route | Purpose |
|--------|-------|---------|
| `GET`  | `/api/config/ssl-info` | Inspect the live served chain as JSON (active set + leaf/intermediate/root + SPKI pins). |
| `POST` | `/api/config/ssl-rotation` | New leaf key for the **currently active** manual set (a or b), under its own CA. Optional `expires_in_days` / `expires_in_minutes`. Errors if set c is active (LE is swap-only). |
| `POST` | `/api/config/ssl-change/{a\|b\|c}` | Swap active set. `a`/`b` manual CAs (expiry allowed), `c` = Let's Encrypt (swap-only). |

### Expiry params (a/b only)

Both optional integers, additive: `total_minutes = days*1440 + minutes`
(default 90 days when both omitted). `<= 0` ⇒ already-expired.

| Body | Meaning |
|------|---------|
| *(omitted)* | 90 days |
| `{ "expires_in_minutes": 2 }` | valid 2 minutes — connect now (works), wait, connect again (fails) |
| `{ "expires_in_minutes": -1 }` | already expired |

> An expired cert fails on TLS date-validation *before* pinning is evaluated, so
> expiry cases test the client's general TLS validity handling — complementary
> to the pinning path. Document which failure you actually observe.

## Test matrix (app pins CA-A intermediate+root, enforce mode)

| Action | Expected | Proves |
|--------|----------|--------|
| `GET ssl-info` | JSON: active set + chain pins | Pins diffable vs app config |
| `POST ssl-rotation` | App **still connects** | CA-level pin survives leaf rotation |
| `POST ssl-change/a` | App **connects** | Baseline / restore |
| `POST ssl-change/b` | App **rejects** (`SSLPeerUnverifiedException`) | Pinning blocks a CA swap |
| `POST ssl-change/c` | Pinned build **rejects**; prod build (pins LE) **connects** | Real LE chain vs pinned CA |
| `ssl-rotation` `expires_in_minutes:-1` | App **rejects** (expired) | Client TLS validity handling |
| `ssl-rotation` `expires_in_minutes:2` | Connects now, **rejects after ~2 min** | Quick live-expiry test |

> To make `change/b` a *fair* pinning test, install CA-B on the device so it's
> otherwise trusted — then rejection is attributable to pinning, not an untrusted CA.

## Guardrails

- **Non-prod only** — gated by `SSL_TEST_ENABLED`; don't set it on real prod.
- **Auth + rate limit + audit log** on every call (see `SslTestController::run`).
- **No shell interpolation** — array command args; `{v}` whitelisted to `a|b|c`;
  expiry params cast to int before reaching the script.
- **Keys off the web root** — everything lives under `/opt/ssl-test`.
